2) I designed preprocessing specifically for NDCG@20

Because the target metric was NDCG@20, I wanted preprocessing that would improve ranking stability and fairness.

Since the data represented positive interactions only, I treated it as binary implicit feedback. If a user interacted with an item, the value was 1; otherwise 0. I removed duplicate interactions within each user row.



b. Per-user stratified splitting

Instead of splitting globally, I created train/validation/test sets per user, so every user contributed interactions to both training and evaluation. This is important for recommendation because otherwise you risk evaluating on users that the model never learned.

I used a structure like:



5 test items per user

1 validation item per user

the rest for training

This gave each user enough held-out positives to compute a meaningful NDCG@20.



c. Warm-test constraint

I made sure held-out test items were “warm,” meaning they appeared often enough globally that the model could actually learn them from training data. This avoided artificially penalizing the model for predicting items it never had a chance to learn.



d. Item filtering / k-core style preprocessing

Because many items were extremely rare, I experimented with keeping only items with at least a small minimum number of interactions, such as 3+ interactions, to reduce noise.



e. Popularity-bias mitigation

The item histogram showed a strong head-tail imbalance, so I explored weighting methods to reduce the dominance of highly popular items. One of the most effective was IDF weighting, which down-weighted very common items and increased the impact of more discriminative interactions.

Graph-based recommendation

Gemini said
tf.GradientTape 是 TensorFlow 框架中用于**自动求导（Automatic Differentiation）**的核心工具。

为了让你更容易理解，你可以把它想象成一台**“记录仪”或“微型录像机”**。

它是怎么工作的？
在深度学习中，模型学习的本质是：做预测 -> 算误差（Loss） -> 求误差对每个参数的梯度（导数） -> 更新参数。

这台“录像机”的作用就在于第三步。它的使用流程通常是这样的：

开机录像：你写下 with tf.GradientTape() as tape:。在这个代码块里面，模型会进行“前向传播”（拿着输入数据去瞎猜一个结果，然后和正确答案比对，算出 Loss）。

记录轨迹：在这期间，tape 会像录像机一样，把你做的所有数学运算（加减乘除、矩阵相乘等）的轨迹都默默记录在案。

倒带求导：跳出这个代码块后，你对它说 tape.gradient(loss, model.trainable_variables)。它就会根据刚才录下的轨迹，利用微积分里的链式法则“倒推”回去，瞬间帮你算出 Loss 对模型每一个权重参数的梯度。

更新参数：你拿到这些梯度后，交给优化器（比如 Adam），优化器就会去调整模型的权重，让下一次的 Loss 变小。

为什么你在 LightGCN 项目中非用它不可？
平时很多初学者写深度学习代码，最后一行直接调用 model.fit() 就完事了。model.fit() 就像是**“自动挡”**，它的底层其实也包裹着 GradientTape，但是它把整个循环封死了，只留了几个固定的参数让你调。

但在你的 LightGCN 项目中，由于是隐式反馈数据（只有点过/没点过，没有具体评分），你需要做动态负采样：针对用户互动过的物品（正样本），你需要随机抽取他们没看过的物品（负样本）来对比 。而且为了防止模型死记硬背，每个 epoch 甚至每个 batch，抽出来的负样本都必须是动态更新的 。

为什么高比例（如 1:10）有好处？：在 38,000 个物品的大库中，用户的正样本极少。如果采用 1:10，模型在一次参数更新中就能看到更多“错误选项”，它能更快地学会区分什么是用户不喜欢的，从而拥有更强的辨别力。实际上，在你们的 NCF 模型中，你们就是用了 1:4 的负采样比例来增强对比信号 。

为什么 LightGCN 最终用了 1:1？

计算瓶颈与内存压力：如果用 1:10，意味着你的 Batch Size 会瞬间膨胀 10 倍。在你们包含 120 万正交互的数据集上，原本的矩阵操作就已经非常吃力了，强行拉高比例会导致显存/内存溢出（OOM），或者每个 Epoch 的训练时间变得极其漫长。


动态采样的“时间换空间”魔法：你们采用了 1:1 的动态随机负采样 。这意味着虽然每个 Epoch 只配对 1 个负样本，但因为有 50 个 Epoch，随着训练的推进，模型其实对每一个正样本都“见识”了 50 个不同的负样本。这种做法既保持了单次计算的轻量（速度快、矩阵小），又通过时间跨度达到了类似于高比例负采样的效果 。
+4

2. tf.GradientTape 的训练代码和 model.fit() 到底有什么不同？
它们的核心区别在于**“控制权的归属”**。

model.fit()（封装好的黑盒/自动挡）：
它假设你的数据是现成不变的。你把 X（正负样本特征）和 Y（标签）塞进去，它内部帮你完成：切分 Batch -> 前向传播 -> 算 Loss -> 反向传播求导 -> 更新权重。


致命缺陷：它极其死板。如果你们想在隐式反馈中“每轮动态刷新负样本”，使用 model.fit() 会非常不灵活。你们在项目中发现，它无法定制化采样过程，导致了运行时问题并阻碍了模型有效学习 。

tf.GradientTape（自定义训练循环/手动挡）：
你完全抛弃了 Keras 的自动循环，自己写了一个 for epoch in range(EPOCHS): 的大循环 。
巨大优势：在每个 Epoch 开始的第一秒，你可以随心所欲地用 NumPy 编写逻辑去重新抽取负样本，组装好最新的、绝不重复的数据，然后再亲自把数据递给模型。你用 GradientTape 仅仅是为了接管“求导”这一个数学动作 。这让你们拥有了对数据输入和梯度更新的绝对控制权 。
+4

3. 这三行代码具体在干什么？为什么 u_pos 要 concatenate 自己？
这三行代码是整个动态采样逻辑的心脏。因为你们放弃了极其拖慢数据加载速度的 BPR 损失（它需要极其复杂的 (u,i,j) 三元组数据结构），转而采用了更高效的点对点（Pointwise）二元交叉熵（BCE）损失 。
+3

BCE 损失把它变成了一个标准的二分类问题：模型只需要判断给定的“一对（User, Item）”是 1（喜欢）还是 0（不喜欢）。
+1

我们逐句拆解：

Python
# 1. 把所有发生过交互的 UserID 复制一遍，首尾相接拼起来
u_batch = np.concatenate([u_pos, u_pos])
解释：u_pos 是所有产生了点击的正样本用户列表（比如 [张三, 李四]）。因为我们要给张三和李四各自准备一个正物品和一个负物品，所以用户列表必须翻倍，变成 [张三, 李四, 张三, 李四]。前半部分用来配对正样本，后半部分用来配对负样本。

Python
# 2. 把真实交互过的正物品，和刚才随机抽出来的负物品，首尾相接拼起来
i_batch = np.concatenate([i_pos, neg_items])
解释：i_pos 是真实点击过的物品（比如 [苹果, 香蕉]），neg_items 是刚用 numpy 随机抽出来用户没见过的物品（比如 [大蒜, 辣椒]）。拼起来后，物品列表变成了 [苹果, 香蕉, 大蒜, 辣椒]。

魔法发生的地方：此时，u_batch 和 i_batch 是严格一一对应的：

位置 0：张三 -> 苹果 (真实交互)

位置 1：李四 -> 香蕉 (真实交互)

位置 2：张三 -> 大蒜 (随机瞎抽的负样本)

位置 3：李四 -> 辣椒 (随机瞎抽的负样本)

Python
# 3. 为上半截的正样本打上标签 1，为下半截的负样本打上标签 0
labels = np.concatenate([np.ones(len(u_pos)), np.zeros(len(u_pos))]).astype(np.float32)
解释：生成一个长度与 u_pos 相同的全 1 数组，再生成一个全 0 数组，拼起来变成 [1.0, 1.0, 0.0, 0.0]。

这样，模型拿到这三组数据后，就能完美地进行二分类学习：它会努力提高 (张三, 苹果) 的预测分（向 1 靠拢），并拼命压低 (张三, 大蒜) 的预测分（向 0 靠拢）。

这就是将复杂的图推荐问题，通过巧妙的 NumPy 数组拼接，降维打击成了一个极度高效的分类问题的全过程。
